## 1. プロジェクト概要

### 1.1 目的・背景

イベント・懇親会などの場で、ホスト（出題者）がリアルタイムにクイズを進行し、複数の参加者がスマートフォンやPCから同時に回答できる「参加型Webクイズアプリ」を構築する。Kahootのような体験を提供しつつ、**その場でのアドリブ出題**や**参加者全員の回答を名前付きで一覧表示**する独自機能を持つ。場の盛り上がりとエンゲージメントを高めることを目的とする。

### 1.2 ターゲットユーザー

| ロール | 説明 |
|--------|------|
| ホスト | クイズを作成・進行する出題者。イベント主催者・幹事など。**メールアドレス＋パスワードでログイン必須** |
| 参加者（学生） | ルームコードで入室して回答するプレイヤー。**学内DBの学生情報で認証する**（認証方式の詳細はwebアプリ「name-roulette」の実装を参照）|

### 1.3 対象プラットフォーム

- Webブラウザ（PC・スマートフォン・タブレット対応のレスポンシブデザイン）
- ネイティブアプリは対象外（MVP時点）

### 1.4 スコープ（MVP / 将来拡張）

**MVP（初期リリース）**
- **ホストはメールアドレス＋パスワードでログイン必須**（アカウント登録・管理機能含む）
- **参加者（学生）は学内DBの学生情報で認証**（認証方式はname-rouletteを参照）
- 事前作成クイズ＋その場でのアドリブ出題の両対応
- 4択形式の問題（将来的に複数形式へ拡張）
- リアルタイム全員同時回答（Kahoot方式）
- 参加者名付き回答一覧表示
- 画像付き問題対応
- カウントダウンタイマー・リアルタイムランキング
- 1セッションあたり最大100人程度

**将来拡張（Could / Won't for MVP）**
- ○×問題・自由記述・複数選択形式
- 動画付き問題
- 結果履歴の保存・再閲覧
- GoogleやLINEなどのSNSログイン追加
- チーム対抗戦モード
- 難易度設定

### 2.2 機能詳細

#### その場での問題入力（アドリブ出題）

ホストはセッション進行中に、次の問題を「その場」で入力できる。問題文・選択肢（2〜4個）・正解・制限時間を入力フォームで設定し、「出題開始」ボタンで全参加者に即時配信する。事前作成クイズとの切り替えはセッション中も可能で、事前作成問題とアドリブ問題を混在させることができる。

#### 参加者名付き回答一覧

各問題の回答締め切り後、ホスト画面（および参加者の結果画面）に「誰がどの選択肢を選んだか」を参加者のニックネームと一緒に一覧表示する。正解者・不正解者がひと目でわかるよう色分けし、会話のきっかけや盛り上がりを演出する。

#### スコアリング

正解かつ回答速度を加味したスコアを算出する（早く正解するほど高得点）。スコアは各問題終了後に集計・ランキング化され、全参加者に同時配信される。

#### セッション管理

ルームコードは6桁の英数字（大文字）で発行される。ホストがセッションを終了するとルームコードは無効になり、参加者は自動的に結果画面へ遷移する。

## 4. システム構成・アーキテクチャ

### 4.1 技術スタック

| 領域 | 技術 | 選定理由 |
|------|------|----------|
| フロントエンド | Next.js 14（App Router）+ TypeScript | SSR・APIルート・型安全性を一括提供。エコシステムが成熟 |
| スタイリング | Tailwind CSS | 高速なUI開発・レスポンシブ対応 |
| リアルタイム通信 | Socket.IO | 自動再接続・ルーム管理・fallback対応。マルチプレイヤーに最適 |
| バックエンド | Node.js（Next.js + カスタムHTTPサーバー） | Socket.IOの永続接続のためカスタムサーバーが必要 |
| **ホスト認証** | **NextAuth.js（Credentials Provider）** | **メール＋パスワード認証。将来のSNSログイン追加も容易** |
| **学生認証** | **学内既存DB（studentsテーブル）を参照** | **認証方式の詳細はwebアプリ「name-roulette」の実装を参照すること** |
| データベース | **学内オンプレ PostgreSQL 16**（ホスト: 172.16.10.11、DB名: students、ユーザー: subamanage） | **既存の学生情報DBをそのまま利用。Supabase不使用** |
| ORM | Prisma | 型安全なDBアクセス・マイグレーション管理 |
| 画像ストレージ | ローカルファイルシステム または 学内ファイルサーバー | オンプレ運用のためクラウドストレージ不使用 |
| **インフラ** | **学内オンプレサーバー（ラズパイ5または既存PC）** | **学内LAN内での運用。外部クラウド不使用。WebSocket永続接続も問題なし** |

> ⚠️ **重要注意**: VercelはWebSocket（Socket.IO）の永続接続を**サポートしていない**。デプロイ先はRailwayまたはRenderを使用すること。

### 4.2 システム構成概要

```
[学生ブラウザ (スマホ/PC)]
         ↕ Socket.IO (WebSocket)
[Next.js カスタムHTTPサーバー + Socket.IOサーバー]  ← 学内オンプレサーバー
         ↕ REST API (Next.js APIルート)
         ↕ Prisma ORM
    [PostgreSQL 16（172.16.10.11 / DB: students）]
         ↑ 学生認証・学生情報参照（studentsテーブル）

[ホストブラウザ]
         ↕ Socket.IO (WebSocket)
[同一サーバー]
         ↕ NextAuth.js セッション管理
    [同一PostgreSQL（ホストアカウント用テーブルを追加）]
```

> ✅ **オンプレ運用のため、Vercelのような外部クラウドは使用しない。WebSocket永続接続の制限も受けない。**

### 4.3 Socket.IOのルーム設計

ルームコードをSocket.IOのroom名として使用する。ホストと参加者は同一roomに参加し、ホストのみが出題イベントをemitできるよう権限を制御する。スコア計算はすべてサーバーサイドで完結させ、クライアントからの改ざんを防ぐ。

### 4.4 外部連携・API

| サービス | 用途 |
|---------|------|
| 学内PostgreSQL（172.16.10.11） | 学生情報の参照・認証、クイズデータの保存 |
| 学内SMTPサーバー | メール通知（app_settingsのSMTP設定を利用） |

## 5. 用語集・データ定義

### 5.1 用語集

| 用語 | 定義 |
|------|------|
| ホスト | クイズを作成・進行する出題者。メールアドレス＋パスワードでログインして利用する |
| 参加者（学生） | 学内DBの学生情報で認証後、ルームコードで入室してクイズに参加するユーザー |
| クイズ（Quiz） | 複数の問題から構成される事前作成の問題セット |
| 問題（Question） | 1つの問い・選択肢・正解・制限時間・画像（任意）の組 |
| アドリブ問題 | セッション中にホストがその場で入力した問題（DBに保存しない） |
| セッション（Session） | ホストがクイズを開始してから終了するまでの1回の実施単位 |
| ルームコード（Room Code） | 参加者がセッションに入室するための一意の6桁英数字コード |
| スコア（Score） | 正解かつ回答速度によって加算されるポイント |

### 5.2 主要エンティティ定義

#### Host（ホストユーザー）

| フィールド | 型 | 説明 |
|-----------|-----|------|
| id | UUID | 主キー |
| email | String | メールアドレス（ログインID・ユニーク） |
| passwordHash | String | bcryptでハッシュ化されたパスワード |
| name | String | 表示名 |
| emailVerified | Boolean | メール確認済みフラグ |
| createdAt | DateTime | 登録日時 |
| updatedAt | DateTime | 更新日時 |

#### Quiz（事前作成クイズ）

| フィールド | 型 | 説明 |
|-----------|-----|------|
| id | UUID | 主キー |
| hostId | UUID | Hostへの外部キー（ログイン済みホストに紐づく） |
| title | String | クイズタイトル |
| createdAt | DateTime | 作成日時 |
| updatedAt | DateTime | 更新日時 |

#### Question（問題）

| フィールド | 型 | 説明 |
|-----------|-----|------|
| id | UUID | 主キー |
| quizId | UUID | Quizへの外部キー |
| order | Int | 問題の順番 |
| text | String | 問題文 |
| options | String[] | 選択肢の配列（2〜4要素） |
| correctIndex | Int | 正解の選択肢インデックス |
| timeLimitSec | Int | 制限時間（秒）。デフォルト20秒 |
| imageUrl | String? | 問題画像のURL（任意） |

#### Session（セッション）

| フィールド | 型 | 説明 |
|-----------|-----|------|
| id | UUID | 主キー |
| quizId | UUID? | Quizへの外部キー（アドリブのみの場合はnull） |
| roomCode | String | 6桁のルームコード（ユニーク） |
| status | Enum | waiting / active / finished |
| currentQuestionIndex | Int | 現在表示中の問題番号 |
| startedAt | DateTime? | 開始日時 |
| finishedAt | DateTime? | 終了日時 |

#### Player（セッション参加者）

| フィールド | 型 | 説明 |
|-----------|-----|------|
| id | UUID | 主キー |
| sessionId | UUID | Sessionへの外部キー |
| studentId | UUID | 学内DBのstudents.idへの参照（認証済み学生を紐づける） |
| nickname | String | 表示名（studentsテーブルのnameを初期値として使用） |
| socketId | String | Socket.IOのsocket ID（再接続時に更新） |
| totalScore | Int | 累計スコア |

> 📌 **学生認証について**: 参加者の認証方式はwebアプリ「name-roulette」の実装を参照すること。学内PostgreSQL（172.16.10.11）の `students` テーブル（student_number / name / email 等）を使って照合を行う。

#### Answer（回答）

| フィールド | 型 | 説明 |
|-----------|-----|------|
| id | UUID | 主キー |
| playerId | UUID | Playerへの外部キー |
| questionId | String | 問題ID（アドリブは一時IDを使用） |
| selectedIndex | Int | 選択した選択肢インデックス |
| isCorrect | Boolean | 正解フラグ |
| responseTimeSec | Float | 回答までの秒数 |
| score | Int | 獲得スコア |
| answeredAt | DateTime | 回答日時 |

#### エンティティ関係

```
【学内既存DB（students DB）】
students ←── Player（studentIdで参照）

【クイズアプリ追加テーブル】
Host 1──N Quiz 1──N Question
           |
           0..1──N Session 1──N Player 1──N Answer
```

## 6. 未決事項・備考

| # | 項目 | 内容 |
|---|------|------|
| 1 | メール確認フロー | 会員登録時にメール確認（本人確認メール）を必須とするか、省略して即利用可能にするか決める必要がある |
| 2 | パスワード要件 | 最低文字数・大文字小文字・記号の要否など、パスワード強度ポリシーを決定する必要がある |

| 6 | ニックネームの扱い | 学生認証後のニックネームをstudents.nameをそのまま使うか、任意変更可能にするかを決める |

| 8 | アドリブ問題の保存 | セッション終了後にアドリブ問題を事後的にホストのクイズとして保存できる機能を追加するか |
| 9 | スコア計算式 | 速度ボーナスの具体的な計算式（例: `1000 × 残り時間 / 制限時間`）は実装時に決定 |
| 10 | 画像アップロード | その場入力時の画像アップロードをどこまでサポートするか（URL入力のみにするか、ファイルアップロードも対応するか） |
| 11 | 同時セッション数 | 同一ホストが複数セッションを同時開催できるかどうか |

### 7.4 推奨設計パターン

Socket.IOのNext.js統合では `server.js` にカスタムHTTPサーバーを作成し、Socket.IOサーバーを同一プロセスで起動する。ルームコードをSocket.IOのroom名として利用し、ホスト権限はサーバーサイドで検証する。スコア計算・正解判定はすべてサーバーサイドで処理し、クライアントからの改ざんを防ぐこと。

## 8. セキュリティ評価・専門家コメント

### 8.1 脅威モデル（主なリスク）

| リスク | 深刻度 | 説明 |
|--------|--------|------|
| スコア・回答の改ざん | High | クライアントが不正な回答やスコアを送信するリスク |
| ルームコードへのブルートフォース | Medium | 6桁コードへの総当たりで他人のセッションに不法入室 |
| 認証情報の漏洩 | High | ホストのパスワードが平文保存・ログ出力されるリスク |
| セッションハイジャック | High | JWTやセッショントークンが盗まれてホストになりすまされるリスク |
| ホスト権限のなりすまし（WebSocket） | High | 認証済みJWTを持たない者がホスト用Socket.IOイベントを送信するリスク |
| 大量接続によるDoS | Medium | 悪意ある大量WebSocket接続でサーバーリソースを枯渇させる |
| 画像アップロードの悪用 | Medium | 不正なファイル形式のアップロードによるサーバー攻撃 |
| XSS（クロスサイトスクリプティング） | Medium | ニックネームや問題文に悪意あるスクリプトを埋め込む |

### 8.2 必須セキュリティ対策

**認証・パスワード管理**
パスワードは必ずbcrypt（コスト係数12以上）でハッシュ化してDBに保存する。平文での保存・ログ出力は絶対に行わない。NextAuth.jsのJWTセッションを使い、有効期限（例: 7日）を設ける。

**ホスト権限制御（WebSocket）**
Socket.IOへの接続時にNextAuth.jsのJWTトークンを検証し、ホストとして認証済みであることをサーバーサイドで確認する。認証されていないクライアントからのホスト用イベント（出題・進行操作）は即座に拒否する。クライアント側のみのチェックは無効。

**スコア・正解判定のサーバー処理**
正解判定・スコア計算は必ずサーバーサイドで行う。クライアントから「正解です」「スコア追加」などのイベントを受け付けない設計にすること。

**入力バリデーション・XSS対策**
ニックネーム・問題文・選択肢はサーバーサイドでサニタイズする。フロントエンドでも表示時にエスケープ処理を徹底する（Reactのデフォルト動作を活用）。

**レート制限**
WebSocket接続・回答送信イベントにレート制限を設ける。ログイン試行にも同様のレート制限を設け、ブルートフォース攻撃を防ぐ。

**画像アップロード制限**
許可するMIMEタイプをjpg/png/gif/webpに限定し、ファイルサイズ上限（例: 5MB）を設ける。Supabase Storageのアクセスポリシーで公開範囲を適切に設定する。

**HTTPS / wss:// の強制**
本番環境では必ずHTTPS/wss://で通信を暗号化する。

### 8.3 推奨セキュリティ対策

- ルームコードのレート制限：同一IPからの入室試行を一定回数で制限し、ブルートフォースを防ぐ
- ルームコードの有効期限：セッション終了後は即座に無効化する
- ルームの参加人数上限：1セッションあたりの最大接続数をサーバーで制限する（例: 200人）
- エラーログ・監査ログ：不審なイベント送信のログを記録し、異常検知に備える

### 8.4 準拠すべき規格・法令

| 規格・法令 | 対応方針 |
|-----------|---------|
| 個人情報保護法 | ホストのメールアドレス・学生の個人情報を扱うため適用対象。プライバシーポリシーの整備が必要 |
| OWASP Top10 | XSS・インジェクション・認証の不備・アクセス制御の不備を中心に対策する |
| HTTPS/TLS | 学内LANのみでの運用であれば省略可だが、将来的な外部公開を考慮して導入を推奨 |
| 既存DBアクセス権限 | subamanageユーザーの権限範囲を確認し、既存の学生情報テーブルへの意図しない変更・削除が起きない設計にする |

### 8.5 セキュリティ専門家からの総合コメント

ログイン機能の追加により、前バージョンで最大リスクだった「ホスト権限の脆弱性」は大幅に改善される。ただし、**認証の導入は新たなリスクも生む**ため、以下の3点を設計段階で必ず組み込むこと。

1. **パスワードの安全な管理** — bcryptによるハッシュ化は必須。平文保存・ログ出力は絶対禁止。パスワードリセットのリンクは短時間（例: 1時間）で失効させる。

2. **JWT × Socket.IO の権限検証** — NextAuth.jsのJWTをSocket.IO接続時にサーバーで検証し、ホスト用イベントへのアクセスを制限する。この検証を怠ると認証の意味がなくなる。

3. **ホストのメールアドレスという個人情報の保護** — メールアドレスを収集する時点で個人情報保護法の適用対象となる可能性が高い。プライバシーポリシーの整備と、メールアドレスのログ出力禁止・不要な第三者提供の禁止を徹底すること。
